# Purpose
Lightweight ADR (Architecture Decision Record) log — why things are the way they are, so nobody "fixes" an intentional design choice by accident.

# Summary
Most consequential decisions: (1) Kernel is HTTP/Kafka-only, never a Spring bean dependency; (2) Liquibase with explicit `include:`, never `includeAll`; (3) reactive end-to-end, JDBC isolated to Liquibase; (4) `@RequirePermission` resolver is pluggable LOCAL/REMOTE/HYBRID so RBAC works before the Kernel ships a permission endpoint.

# Details

## ADR-001 — Kernel consumed only via HTTP/WebSocket + data-type imports
**Status**: Accepted (foundational, CLAUDE.md).
**Decision**: TiiBnTick Core never injects Kernel-internal Spring beans/interfaces directly (except explicitly-published SPI interfaces like `ReactivePermissionResolver`, `RoleRepository`). All Kernel interaction is HTTP (`kernelWebClient`) or Kafka, plus importing Kernel **data types** (`TenantId`, `Money`, etc.).
**Why**: Kernel is owned by a different team/repo (TSAFACK Savio) and evolves independently — coupling to its internal beans would break on every Kernel release.
**Consequence**: Kernel role-provisioning currently gets `404` (endpoint not implemented yet) — this is expected, not a bug. See `knowledge/known-issues.md`.

## ADR-002 — Explicit Liquibase `include:`, never `includeAll`
**Status**: Accepted, enforced 2026-06 after a multi-day debugging session.
**Decision**: Root master changelog (`tnt-core-master.yaml`) explicitly `include:`s one uniquely-named master changelog per module. No module changelog uses `includeAll`.
**Why**: `includeAll` scans the entire JVM classpath (`ClassLoader.getResources()`), not the current module's JAR — with ~25 module JARs simultaneously on `tnt-bootstrap`'s classpath, it silently picks up the wrong files or no-ops on name collisions. This caused ~8 previously-undetected migration bugs to ship silently.
**Consequence**: Adding a new module's migrations requires a manual `include:` line in `tnt-core-master.yaml` — easy to forget (see ADR-004 below, the `tnt-sync-core` incident).

## ADR-003 — Reactive end-to-end; Liquibase is the one blocking exception
**Status**: Accepted (foundational).
**Decision**: All runtime DB access is R2DBC (non-blocking). Liquibase uses a separate, blocking JDBC driver, scoped to schema migration only — enforced by `maven-enforcer-plugin` banning `org.postgresql:postgresql` at compile scope.
**Why**: Liquibase has no mature reactive driver; isolating it to `runtime` scope + migration-only usage keeps the app's hot path 100% non-blocking.

## ADR-004 — Pluggable permission resolution (LOCAL/REMOTE/HYBRID)
**Status**: Accepted 2026-06-29, implemented in `tnt-roles-core`.
**Decision**: `@RequirePermission`'s public API never changes. Behind it, `ReactivePermissionResolver` has three interchangeable implementations selected by `tnt.roles.permission.mode`: `LocalReactivePermissionResolver` (resolves from local `UserRoleAssignmentRepository`/`RoleRepository`/`TntRoleDefinitionRegistry`), `RemoteReactivePermissionResolver` (Kernel HTTP, forward-compatible), `HybridReactivePermissionResolver` (local-first, Kernel fallback). All wrapped by a Caffeine-backed `CachingReactivePermissionResolverDecorator`, with Kafka-driven cache invalidation (`tnt.roles.permission-changed` topic, currently unused by any producer — scaffold for future role-mutation flows).
**Why**: The user required RBAC to be **fully functional today**, without disabling AOP or using an always-allow resolver, even though the Kernel doesn't expose a permission-resolution REST endpoint yet.
**Consequence**: Default mode is `LOCAL`. Switching to `HYBRID`/`REMOTE` requires zero code changes — just `tnt.roles.permission.mode=HYBRID`.

## ADR-005 — Module master changelog filenames must be globally unique
**Status**: Accepted (consequence of ADR-002).
**Decision**: No module changelog is named `db.changelog-master.yaml`; each is `tnt-<module>-master.yaml`.
**Why**: With every module JAR on one classpath, a generic name is ambiguous to Liquibase's classpath resolver.

## ADR-006 — Lombok and springdoc-openapi pinned to Spring Boot 4 / Framework 7 / JDK 25-compatible versions
**Status**: Accepted 2026-06-30.
**Decision**: `lombok.version=1.18.46` (was 1.18.32 — broke javac on JDK 25 with `TypeTag :: UNKNOWN`), `springdoc.version=3.0.3` (was 2.5.0 — Boot-3-only, broke `/swagger-ui.html`).
**Why**: This repo runs ahead of most libraries' stable support matrix (Spring Boot 4.0.6 / Framework 7 / JDK 21-25). Pin versions explicitly rather than relying on transitive resolution; re-verify after every Spring Boot bump.
**Consequence**: `RT-comops-kernel-core` transitively pulls non-jakarta `swagger-annotations`/`swagger-models:2.2.22`, which conflicts with springdoc 3.x's jakarta variants — excluded explicitly in root `pom.xml` `dependencyManagement`, plus an explicit `swagger-annotations-jakarta` dependency added globally so modules using `@Operation`/`@Tag` compile correctly. See `knowledge/known-issues.md`.

## ADR-007 — `tnt-billing-dsl` and `tnt-accounting-core` use `infrastructure/` + `domain/port` instead of standard hexagonal
**Status**: Accepted (foundational, CLAUDE.md).
**Decision**: These two modules use `domain/port/{in,out}` for use-case/port interfaces and `infrastructure/{adapter,config,persistence}` for the outward-facing layer, instead of `application/port` + `adapter/`.
**Why**: Same dependency direction, just renamed for domains where the "domain service" (rule evaluator, double-entry bookkeeping) needs first-class status alongside the aggregate.

## ADR-008 — `tnt-media-core` stays on local MinIO storage; not migrated to the Kernel `file-controller`
**Status**: Accepted 2026-07-08, decided as module 1 of the "Core as Kernel facade/BFF" initiative (see `docs/kernel-api/`).
**Decision**: `tnt-media-core`'s generic object storage (upload/download/presigned URLs, SHA-256 dedup, image processing, type-based expiry + cleanup scheduler) and the incident-evidence WORM archival (`IncidentMediaStorageAdapter`, 7-year legal retention via MinIO Object Lock) remain local, backed by MinIO. QR code generation, digital signature capture, and PDF manifest/receipt rendering (`QrCodeService`, `DigitalSignatureService`, `ManifestPdfService`, `ReceiptGenerator`) also remain local. None of this is delegated to the Kernel.
**Why**: The Kernel's only relevant surface is `file-controller` (`POST /api/files`, `GET /api/files/{id}`, `/content`, `/metadata`, `/review`, schema `StoredFileResponse`). It is materially narrower than what TiiBnTick's media platform needs: no delete endpoint (files accumulate forever — no expiry/cleanup possible), no presigned-URL support (every read would have to proxy binary content through Core, adding a hop to every image/PDF fetch), no bucket/tenant-isolation concept, and no retention/legal-hold semantics at all (the WORM 7-year requirement for incident evidence, a Cameroon legal-compliance constraint, has no Kernel equivalent — moving to it would be a compliance regression, not a simplification). QR generation, digital-signature capture, and PDF rendering are pure TiiBnTick domain algorithms; the Kernel's PDF-producing endpoints (payroll documents, accounting rapports) are narrow and domain-specific to Kernel's own workflows, not a generic rendering service TiiBnTick could call for delivery manifests/receipts.
**Consequence**: This is a deliberate exception to "maximize Kernel consumption" — adopting the Kernel API here would remove capabilities TiiBnTick production traffic depends on. Re-evaluate only if the Kernel's `file-controller` grows delete/presigned-URL/retention support. See `docs/kernel-api/endpoints.md` (`file-controller` section) for the current (2026-07-08) Kernel file API surface.

## ADR-009 — `tnt-resource-core` fleet management (Vehicle/Equipment/FreelancerVehicle) stays local
**Status**: Accepted 2026-07-08, module 2 of the "Core as Kernel facade/BFF" initiative.
**Decision**: `Vehicle`, `Equipment`, `FreelancerVehicle`/`FreelancerEquipment`, `ResourceAllocation`, `VehicleMaintenanceRecord`, and the OR-Tools-backed matching engine (`VehicleMatchingApplicationService`/`FindBestVehicleForMissionUseCase`) stay local, R2DBC-persisted. The existing `KernelActorAdapter` (fail-open existence/active check against the Kernel actor when registering a freelancer's vehicle/equipment) is kept and is sufficient — no further delegation.
**Why**: The Kernel does expose a generic `resource-controller` (`/api/resources`, register/assign/reserve/maintenance/location-observation) that looks superficially similar, but its schema (`MaterialResourceResponse`: resourceCode, name, freeform `category` string, serialNumber, status, lat/long, IP/MAC address) is shaped for IoT/network-asset tracking, not fleet management — no capacity, fuel type, VIN, odometer, or vehicle/equipment type taxonomy, and `MaintenanceRecordResponse` is far thinner than TiiBnTick's typed maintenance model. There is no Kernel equivalent of the OR-Tools vehicle-matching engine at all. Moving fleet persistence to Kernel HTTP calls would also add a network round-trip to a real-time dispatch path (vehicle GPS updates go through Redis today, not any REST call) and break every other module's direct DB-level `VehicleRepository`/`EquipmentRepository` port usage.
**Consequence**: Same pattern as every other already-migrated business/identity module in this repo — Kernel is used only as a cross-reference existence check, real domain data and logic stay local. While reviewing `KernelActorAdapter`, found and fixed a real bug (missing `/api` prefix on `GET /actors/{id}` → `/api/actors/{id}`) — see `docs/knowledge/known-issues.md` #10.

## ADR-010 — `tnt-product-core`: real catalog-duplication found, migration path documented but NOT executed this pass
**Status**: Proposed 2026-07-08, module 3 of the "Core as Kernel facade/BFF" initiative — deliberately not implemented yet (see Consequence).
**Finding**: This is the strongest genuine duplication found so far. The Kernel's `product-controller` + `product-catalog-controller` + `product-structure-controller` already provide full CRUD for exactly the commerce fields TiiBnTick's local `Product` aggregate also owns — `sku`, `name`, `categoryCode`/`categoryId`, `unitPrice`/`basePrice`, `currency`, `status`, `cost`, `photo`, `uom`, plus variants (with attributes and time-effective pricing via `GET .../prices/effective`), batches, specs, and category translations — none of which TiiBnTick's local model has (no batch tracking, no time-effective pricing, no i18n categories today). The domain code already anticipated this: `Product.catalogProductId` is documented as "Kernel integration key… logical reference only", with backward-compatible factory overloads for the pre-Kernel-link shape. Today it's write-optional and never authoritative: `ProductApplicationService.createProduct` only *validates* `catalogProductId` if the caller happens to supply one (fail-open, `existsAndActive`), and TiiBnTick's own `sku`/`name`/`price`/`category` are always the values actually saved and served — never sourced from the Kernel.
**Why not migrated now**: Making the Kernel the authoritative source for these fields (fetch-through or sync-on-write, with `Product`'s local persistence shrinking to just the logistics-specific extension — `weightKg`, `dimensions`, `logisticsProfile`, `imageKeys`) is a real schema-and-data migration, not a refactor: it needs (1) a backfill strategy for existing local-only products that have no Kernel counterpart yet, (2) a decision on read consistency (proxy every read through the Kernel vs. sync/cache locally and handle drift), and (3) a check on `tnt-inventory-core`/`tnt-sales-core`, which reference `productId` — need to confirm they don't also cache denormalized name/price locally, or the duplication just moves rather than disappears. This is exactly the kind of change ADR-004 calls out as needing to be deliberate rather than done inline.
**Recommendation for a follow-up session**: (1) make `catalogProductId` mandatory on create, calling `POST /api/products` in the Kernel first and using its returned id; (2) turn `KernelProductPort`/`KernelProductAdapter` into a real read port (`findByCatalogProductId` already exists and is fixed — see known-issues #10b) that `GetProductUseCase`/`ListProductsByTenantUseCase` merge with the local logistics-extension row; (3) Liquibase changeset to drop the now-redundant commerce columns from `product` once backfill is verified.
**Consequence**: No code behavior changed by this ADR beyond the `/api` prefix fix in `KernelProductAdapter` (see known-issues.md #10b) — `Product` keeps its full local commerce fields for now. Re-visit as a dedicated module-3-follow-up, not silently dropped.

## ADR-011 — `tnt-inventory-core`: `KernelInventoryPort` premise doesn't match the Kernel's inventory model; dead code removed, live method left honestly broken
**Status**: Accepted 2026-07-08, module 4 of the "Core as Kernel facade/BFF" initiative.
**Decision**: Removed `KernelInventoryPort.findByKernelStockEntryId`/its `KernelInventoryAdapter` implementation entirely (confirmed zero callers anywhere in the codebase). Kept `findByProductAndWarehouse` with its existing fail-open no-op behavior unchanged, but documented in-code why it cannot be correctly fixed without further work — did not wire it to the wrong Kernel identifiers just to make it "look" fixed.
**Why**: `StockEntry`/`InventoryMovement`/`InventoryAlert` (movement ledger, low-stock alerting, reorder thresholds) are legitimate local TiiBnTick logistics logic with no Kernel equivalent — kept local, same conclusion as every other module so far. But `KernelStockEntryDto`'s premise — a stable, fetchable-by-id "stock entry" in the Kernel — doesn't exist: the Kernel's `inventory-movement-controller` models stock as a computed balance over an append-only movement ledger (`GET /api/inventory/movements/balance?organizationId&agencyId&productId` → `StockBalanceResponse{organizationId,agencyId,productId,onHandQuantity}`), not a persisted, individually-addressable entity. `findByKernelStockEntryId` had zero callers — genuinely dead code, safe to delete outright (mission step 4: "supprimer toute logique devenue inutile"). `findByProductAndWarehouse` is still called (`InventoryApplicationService.resolveOrCreateStockEntry`), but correctly implementing it requires a Kernel `organizationId`/`agencyId`, and this module has no `Warehouse` domain concept or resolution path from its local `warehouseId`/`tenantId` to those Kernel ids — inventing one would be a guess, not a fix.
**Consequence**: `findByProductAndWarehouse` keeps calling a non-existent path (same net runtime behavior as before — always empty, fail-open) rather than being wired to a real endpoint with guessed, possibly-wrong parameters. Follow-up needed before this can genuinely work: (1) establish how `warehouseId` maps to a Kernel `agencyId` (likely via `tnt-organization-core`'s Agency, given `tnt-inventory-core` is L4 and can legitimately depend on L2 concepts), (2) thread `organizationId` through `ReceiveStockCommand`/`TntSecurityContext`, (3) switch the call to `GET /api/inventory/movements/balance`.

## ADR-012 — Central `KernelResponses` helper; the Kernel wraps every response, and no adapter was unwrapping it
**Status**: Accepted 2026-07-08, cross-cutting fix applied while reviewing modules 2–6.
**Finding**: Verified live against `kernel-core.yowyob.com` (not just from docs) — every Kernel endpoint wraps its payload as `{success, data, message, errorCode, timestamp}` (matches `KernelApiEnvelope` already used correctly by `tnt-auth-core`'s `KernelAuthGatewayAdapter`, and every `ApiResponseXxx` schema in `docs/kernel-api/schemas.md`). Every other `Kernel*Adapter` in the codebase deserialized the raw response body directly into a flat DTO (`.bodyToMono(SomeDto.class)` / `.bodyToFlux(SomeDto.class)`), skipping this envelope entirely. Combined with the universal fail-open error handling, this meant these adapters have never been able to return real data even when the Kernel responded successfully (200, `success:true`) — Jackson would bind an object with all fields null/default rather than throw, so the bug was completely invisible.
**Decision**: Added `KernelEnvelope<T>` + `KernelResponses` (`foundation/tnt-common-core`, package `com.yowyob.tiibntick.common.kernel`) as the one shared place every Kernel adapter unwraps envelopes and applies fail-open error handling — `unwrapObject`/`unwrapList` (swallow 404 + all errors → empty, the default contract) and `unwrapObjectOrPropagate` (unwrap only, let the caller keep its own error contract — needed by `tnt-organization-core`/`tnt-tp-core`, whose existence checks are intentionally hard-blocking on non-404 errors). `spring-webflux` + `jackson-annotations` added to `tnt-common-core`'s POM since `WebClient` and `@JsonIgnoreProperties` weren't previously needed there. Satisfies the mission's "centraliser les appels REST" requirement directly.
**Applied to**: `KernelActorAdapter` (tnt-resource-core), `KernelProductAdapter` (tnt-product-core), `KernelOrganizationAdapter` (tnt-organization-core), `KernelThirdPartyAdapter` (tnt-tp-core), `KernelPermissionAdapter`/`KernelRoleAdapter` (tnt-administration-core), `KernelRoleProvisioningAdapter` (tnt-roles-core), `KernelNotificationClient` (tnt-notify-core), `KernelSalesOrderAdapter` (tnt-sales-core), `KernelAccountingAdapter`'s `findInvoiceById` (tnt-accounting-core). Deliberately **not** applied to calls that hit a non-existent or wrong-shaped Kernel resource (`KernelInventoryAdapter.findByProductAndWarehouse`, `KernelAccountingAdapter.findJournalEntryById`) — see ADR-011/013; unwrapping an envelope around a call to the wrong URL doesn't fix anything and would read as more "done" than it is.
**Second-order finding — DTO field shapes**: while fixing this, found several `KernelXxxDto`s also assume JSON field names/shapes that don't match the real Kernel schema (e.g. `id` vs `kernelXxxId`, `customerThirdPartyId` vs `clientThirdPartyId`, an `isActive`/`active` boolean the Kernel doesn't model — it uses a `status` string instead). Fixed where cheap and schema was already in hand this session (`KernelSalesOrderDto`, `KernelInvoiceDto`, via `@JsonProperty` aliases). **Not** exhaustively re-verified for every DTO (`KernelActorDto`, `KernelOrganizationDto`, `KernelThirdPartyDto`, `KernelProductDto`, `KernelPermissionDto`, `KernelRoleDto`) — see known-issues.md #12b. Recommend a dedicated pass cross-checking each remaining DTO against its real `docs/kernel-api/schemas.md` entry before relying on these checks for anything beyond "did the Kernel respond at all."

## ADR-013 — `tnt-accounting-core`: envelope/path fixes applied; chart-of-accounts/journal-entry duplication flagged, not executed
**Status**: Accepted 2026-07-08, module 6 of the "Core as Kernel facade/BFF" initiative.
**Decision**: Fixed `KernelAccountingAdapter.findInvoiceById` (real resource, `/api/accounting/invoices/{id}`, unused today but kept). Left `findJournalEntryById` calling a non-existent path deliberately — the closest Kernel candidate, `GET /api/accounting/operations/{id}` → `AccountingOperationView` (single `amount`, `operationType`, no debit/credit split), doesn't match TiiBnTick's double-entry `JournalEntry` shape (separate debit/credit totals) closely enough to map without guessing. Fixed the URL prefix on `findInvoiceByReferenceId` but left its query broken — the Kernel's `GET /api/accounting/invoices` only filters by `organizationId`, not `tenantId`/`referenceId`, and this module has no `tenantId`→Kernel-`organizationId` resolution (same class of gap as ADR-011's warehouse→agency mapping).
**Bigger finding, deliberately not acted on**: the Kernel's `chart-of-accounts-controller` (`/api/accounting/accounts`, including `POST .../generate`) and the broader `accounting-controller`/`accounting-report-controller` family (balance sheet, compte-résultat, grand-livre, invoices, journals — a materially complete double-entry bookkeeping + financial-reporting engine) look substantially more capable than TiiBnTick's local `Account`/`JournalEntry` aggregates. Whether TiiBnTick's local ledger should become a thin view over Kernel's accounting engine is a real, high-stakes architecture question — **not decided or executed here**, given financial/audit data integrity implications and the need for stakeholder input on compliance requirements (OHADA reporting obligations, reconciliation guarantees). `JournalBalanceValidator` (a synchronous, in-memory double-entry invariant check, no I/O) stays local regardless of that decision — it's a fast-fail validation, not a duplicated service, and delegating it to an HTTP call would be a pure regression. `OhadaChartOfAccountsInitializer` seeds a curated, TiiBnTick-logistics-specific subset of accounts (not a generic full OHADA plan) — legitimate to keep even if the Kernel can also generate accounts.
**Consequence**: Follow-up needed, explicitly out of scope for this pass: decide TiiBnTick's accounting module's long-term relationship to the Kernel's accounting engine before doing anything with `Account`/`JournalEntry` persistence.

## ADR-014 — `tnt-actor-core`: new `IKernelActorPort`/`KernelActorAdapter` (module had zero Kernel integration)
**Status**: Accepted 2026-07-08, module 7 (first identity-layer module) of the "Core as Kernel facade/BFF" initiative.
**Decision**: Added `IKernelActorPort` (application/port/out), `KernelActorDto` (domain/model, field names matching the Kernel's real `ActorResponse` schema exactly — no aliasing needed), and `KernelActorAdapter` (adapter/out/kernel), built on `KernelResponses` (ADR-012) from the start. Wired a fail-open `kernelActorPort.exists(actorId)` check into all four profile-creation services (`ClientProfileService`, `FreelancerService`, `DelivererService`, `RelayOperatorService`) — logs a warning and proceeds if the Kernel actor isn't found/reachable, never blocks profile creation. Registered in `ActorCoreConfig`'s `@ComponentScan`.
**Why**: every `TntActorProfile` subtype (`ClientProfile`, `FreelancerProfile`, `DelivererProfile`, `RelayOperatorProfile`) requires a non-null `actorId` by construction — it's explicitly an extension of a Kernel actor, yet nothing validated that actor actually existed. This was the only module in the mandated business+identity list with *zero* Kernel touchpoint.
**Why fail-open, not hard-blocking** (unlike `tnt-organization-core`/`tnt-tp-core`'s pattern for their own root entity): the Kernel's `actor-controller` doesn't document a single-actor-by-id GET endpoint at all (same gap as ADR-009/known-issues.md #10) — hard-blocking against a call that may permanently 404 would risk blocking all actor-profile creation in production. Revisit once the Kernel confirms/adds that endpoint.
**Kernel's `Actor` has no active/status field** (per `ActorResponse` in `docs/kernel-api/schemas.md`) — unlike other Kernel entities, existence is the only signal available, so the port exposes `exists()`, not `existsAndActive()`, to avoid promising a check that can't be performed (the exact class of bug found and fixed under ADR-012/#12b elsewhere).
**Consequence**: compiles clean, all 39 pre-existing tests + 3 new adapter tests pass. No hard-blocking behavior change to any existing creation flow — purely additive enrichment.

## ADR-015 — `tnt-organization-core` and `tnt-tp-core`: hard-blocking Kernel checks confirmed correct design, envelope bug fixed
**Status**: Accepted 2026-07-08, modules 8–9 of the "Core as Kernel facade/BFF" initiative.
**Decision**: No architectural change. `KernelOrganizationAdapter.findById` (validates a Kernel Organization before creating an Agency/Branch/HubRelais/FreelancerOrg) and `KernelThirdPartyAdapter.findById` (validates a Kernel ThirdParty before creating a `TntClientProfile`) both correctly propagate non-404 errors rather than fail-open-swallowing them — the one deliberate exception to this repo's usual fail-open pattern, and the right call here: these local aggregates are direct extensions of *their own* Kernel counterpart (not an optional cross-reference to a foreign domain, unlike e.g. `tnt-resource-core`'s vehicle→actor check), so a Kernel that's reachable but says "doesn't exist" should genuinely block creation.
**Fixes applied**: `KernelOrganizationAdapter`'s path had an extra `/v1/` segment (`/api/v1/organizations/{id}` → `/api/organizations/{id}`, confirmed against `docs/kernel-api/endpoints.md`) — a real bug, now fixed. `KernelThirdPartyAdapter`'s path was already correct (`/api/third-parties/{id}` — a stale inline comment speculated a wrong `/v1/` alternative; removed the risk by leaving the correct constant as-is). Both adapters migrated to `KernelResponses.unwrapObjectOrPropagate` (ADR-012) to fix the envelope bug while preserving their existing non-fail-open error contract exactly.
**Consequence**: `Agency`/`Branch`/`HubRelais`/`FreelancerOrg` (organization-core) and `TntClientProfile`/rating/loyalty/KYC extensions (tp-core) all stay fully local — legitimate TiiBnTick-specific extensions over a thin, now-correctly-implemented Kernel existence gate. No further migration warranted for either module.

## ADR-016 — `tnt-administration-core`: `TntRoleTemplateRegistry` stays local — it's the source pushed to the Kernel, not sourced from it
**Status**: Accepted 2026-07-08, module 10 (final module) of the "Core as Kernel facade/BFF" initiative.
**Decision**: No migration. `TntRoleTemplateRegistry`'s ~20 static role templates (9 canonical `TntRole`-aligned + operational/FreelancerOrg templates) stay exactly as-is.
**Why**: this isn't duplicated Kernel data — it's TiiBnTick's own RBAC taxonomy (permission codes like `delivery:track`, `freelancer:mission:take`, `point-relais:receive`, `blockchain:wallet:read` are entirely TiiBnTick vocabulary, meaningless to the generic Kernel), and the *direction of truth* already runs the right way: `ProvisionTntRoleTemplatesUseCase` **pushes** these templates to the Kernel via `KernelRoleProvisioningAdapter.provisionRole` (`POST /api/roles`), storing the Kernel-assigned `kernelRoleId` back onto `TntRoleDefinition` as a link — the reverse of what "delegate to Kernel" would mean. `KernelPermissionAdapter`/`KernelRoleAdapter` (read-side, used elsewhere in the module) and `KernelRoleProvisioningAdapter.roleExists` (write-side idempotency check) were both broken by the envelope bug (ADR-012) — `roleExists` always returned `false`, so every tenant onboarding silently re-POSTed all ~20 roles and relied on the Kernel's `409 CONFLICT` to no-op rather than skipping via the intended existence check. Now fixed: `roleExists` correctly short-circuits already-provisioned roles.
**Consequence**: this closes out the initiative's mandated module list (see `docs/memory/completed.md` for the full session summary). No code changes beyond the already-applied ADR-012 envelope fix were needed for this module.
